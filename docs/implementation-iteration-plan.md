## 6. 当前执行阶段展开

### M6 主链路联调与验收

- 所属方向：`联调`
- 目标：
  - 基于已完成的前后端能力开展真实任务流联调与缺陷清单收口
  - 验证窗口、托盘、任务列表、详情抽屉、设置页与接入提示的完整主链路
  - 形成最终验收记录、剩余风险和回归结论
- 为什么先做这一阶段：
  - `F5` 已完成前端体验收口，托盘最小化、窗口状态同步、全局提示降噪与任务行内反馈已具备联调基础
  - 当前剩余工作不再是单域编码，而是围绕真实后端和 Electron 运行态做最终验收
- 对应设计依据：
  - 前后端详细设计中的主链路流程、任务状态流转和接口契约约束
  - 前端详细设计中的桌面端整体体验、状态提示和窗口操作要求
  - 架构文档中 Electron 壳层与本地服务联动的边界定义
- 具体动作：
  1. 联调任务创建、列表刷新、详情操作、设置保存、剪贴板提示、外部接管提示全链路
  2. 验证 Electron 托盘、窗口恢复、最小化与本地服务运行态表现
  3. 汇总缺陷清单、已知风险与回归结果
  4. 形成最终验收记录并准备收尾
- 需要修改 / 新建的主要文件：
  - 以联调记录和问题修复为主，必要时联动 `desktop-app/**` 与 `local-service/**`
- 不应改动的范围：
  - 浏览器扩展内部实现细节
- 产出物：
  - 主链路联调结果
  - 缺陷清单与处理结论
  - 验收记录与剩余风险说明
- 验收标准：
  - 核心流程可走通
  - 存在问题有清晰记录和回归结论
  - 可作为当前迭代的交付版本
- 完成后如何更新计划状态：
  - `M6` 改为 `已完成`
  - 当前状态切换为 `已完成`
  - 状态维护记录补充最终验收结果

## 9. 风险、阻塞与回退策略

- 当前最大阻塞项：
  - 暂无功能性阻塞，当前已转入主链路联调与验收阶段
- 阶段依赖风险：
  - `M6` 依赖真实 Electron 运行态与本地服务联动验证；若缺少手工回归，窗口托盘与接入提示仍可能残留边界问题
- 人力 / 资源风险：
  - 单人开发，两周窗口内不适合追加新范围
- 技术验证风险：
  - 前端 `SSE` 与长列表刷新性能
  - Electron 壳层已完成一次启动验证，但后续仍需结合真实任务流持续回归窗口与托盘交互
  - Java 8 控制台中文日志显示乱码仍待后续统一处理，但不影响当前功能联调
- 回退策略：
  - 若 `B4 aria2 适配` 超预期，可先保留最小下载能力，延后高级状态同步
  - 若 `F4 capture` 超预期，可先保证手动创建任务链路，接管提示做降级
  - 若整体时间不足，优先保证 `B1-B5 + F1-F3 + M6`，其余放入后续迭代
